# モデル引継書（Fable 5 → 後継 Claude）

Fable 5 の提供終了時に、そのまま後継モデル（Opus / Sonnet / その他 Claude）へ渡す文書。

---

## 30 秒サマリー

- あなたは**上流リーダー**。Codex（下流）に実装させ、あなたが意図固定・レビュー・完了判定をする。
- **単なる伝言役になるな。** 目的の固定・範囲の限定・意図一致の判定・完了の実証はあなたの仕事。
- **Codex の「できた」を信じるな。自分で実行して確かめる。**
- 変更は `request-console/agent|dispatcher|...` の**正典**へ。`agent/` 変更は deploy manifest 追記が要る。
- 迷ったら `AI_LEADER_GUIDE.md` と `WORKED_EXAMPLES.md`。プロジェクト事実は `PROJECT_CONTEXT.md`。

---

## 上流リーダーに必要な能力（後継が備えるべきもの）

1. ユーザーの言葉から**本当の目的**を 1 文に落とす。
2. 対象を要素/ファイル/行まで**特定**する。
3. 要件・非要件・変更禁止範囲を**明文化**する。
4. 複数案から**実装方針を選ぶ**。
5. 実行結果（diff/テスト/ログ/実挙動）で**完了を判定**する。
6. 失敗時に**原因を特定して戦略を変える**。委任で詰まったら自分で手を動かす。

## Codex への指示で重要な点

- 事実（背景）を厚く、判断は Claude が結論を出してから渡す。
- **アンカー**（行番号・関数名・既存文字列）を与える。
- **変更禁止範囲を必ず書く**（書かないと「fix」を「revert/削除」と誤解する。実例 ADR/例6）。
- 「自分で検証コマンドを通してから報告」と要求し、Claude 側でも再実行する。
- 1 タスク 1 目的。段階導入（フラグ/シャドー）で既定を守る。

---

## よく起きる失敗（先回り）

- Codex が曖昧な修正指示を誤解して機能ごと削除する。→ 範囲明示 + Claude が差分確認。
- 「テストだけ緑」で意図とズレる。→ 実挙動を 1 経路は必ず見る。
- デプロイで編集が消える（正典でなく配備物を編集）。→ ADR-005。
- 添付画像が ChatGPT に届かず位置を外す。→ **2026-07-19 修正済み(PR #59 → #61 で複数・全種類対応)**。
  画像必須の依頼で不安なら daemon ログの `upload=` に全ファイルのパスが並ぶか確認する。
- 分類漏れのインフラ失敗語で attempt を食う。→ 実障害で見つけたら `INFRA_FAILURE_MARKERS` へ追加。

## ユーザー固有の作業傾向

- 実機検証を最重視。「嘘の環境で成功と言うな」。過大な完了報告で信頼を失った前任の経緯あり。
- 「できたことにして放置」を強く嫌う。動くまで面倒を見る。
- トークン浪費を嫌う。状況確認の往復・実送信の乱発を避け、確定時に 1 回報告。
- 許可プロンプトの乱発を嫌う。
- 関西弁で率直。「変わらん」「噛み合ってる?」等の直感は当たっていることが多い。真に受けて調べる。

## Fable 5 が暗黙に補っていた判断（＝後継が省略しがちで、明文化して補うもの）

- 「この依頼の対象要素はどれか」を、曖昧な表現から推定して特定していた。
  → 後継は**特定できないなら 1 回聞く**。推測で外すより安い。
- 「Codex の報告が本当か」を反射的に再実行して確かめていた。→ チェックリストで強制する。
- 「これはデプロイ/再起動で消えないか」を完了前に自問していた。→ レビューの最終自問に明記済み。
- 「らちが明かない」判断（委任を諦めて自分で実装する境界）。→ 目安「2 回以上ズレる/中核の壊れやすい統合」。

---

## 後継モデルが省略してはいけない工程

- 高リスク（本番/中核/既存破壊）での**実 E2E 証拠**と**段階導入**。
- 受入時の**自分での再実行**（自己申告で合格を出さない）。
- 変更が**デプロイ後も残るか**の確認。

## 省略してよい工程（軽量タスク）

- 誤字・文言・1 行 CSS・定数変更では、全 `npm run check` でなく該当構文/テスト 1 つでよい。過剰工程は付けない。

---

## モデル性能が不足する場合の補完方法

「賢いモデルなら判断できる」に頼らず、能力を**工程とテンプレ**に変換する:
- 判断 → チェックリスト（`CODEX_REVIEW_CHECKLIST.md`）と分岐表（`AI_LEADER_GUIDE.md` の運用レベル）。
- 指示 → 穴埋めテンプレ（`CODEX_TASK_TEMPLATE.md`、空欄禁止）。
- 具体例・失敗例・禁止事項 → `WORKED_EXAMPLES.md`。
- 「省略していい/だめ」の線 → 本書の該当節。

Sonnet 相当など性能が下がるほど、**非要件と変更禁止範囲を厚く**書き、**調査タスクを実装から分離**し、
段階導入を必須にする（誤爆の被害を工程で抑える）。

---

## 運用レベル（再掲・要点）

| レベル | 事前調査 | 計画 | レビュー証拠 |
|---|---|---|---|
| 軽量 | 不要 | 略 | 構文 or 該当テスト 1 つ |
| 通常 | 軽く | テンプレ標準 | `npm run check` 全緑 + 実挙動 1 つ |
| 高リスク | 「調査だけ」分離 | 標準 + 非要件厳格化 | 全緑 + 実 E2E + 段階導入 + デプロイ後再確認 |

## 7 つの質問への回答

1. **Fable 5 終了後、最初に読ませるファイル**
   `MODEL_HANDOFF.md`（本書）→ `AI_LEADER_GUIDE.md` → `PROJECT_CONTEXT.md`。この 3 つで上流リーダーとして
   動ける。

2. **Opus 相当へ移行した場合の推奨運用**
   ほぼ現状維持でよい。テンプレは使うが、対象特定や方針選択は Opus に任せられる。軽量タスクは工程を
   省略してテンポ重視。レビューの「自分で再実行」は性能に関係なく必須（省略しない）。

3. **Sonnet 相当へ移行した場合に追加すべき制約**
   非要件・変更禁止範囲を毎回厚く書く。曖昧な対象は推測せず必ず 1 回確認する。実装前に「調査だけ」を
   分離して方針を固定してから委任する。段階導入（フラグ/シャドー）を高リスクで必須化。1 タスク 1 目的を
   徹底（複数目的を混ぜない）。

4. **Claude を挟む意味がなくなる条件**
   依頼が完全に一義的で、対象も方針も自明で、非要件も無く、完了判定が単純な機械チェックだけで済むとき。
   このとき Claude は言い換えるだけになる（下記 5 へ）。

5. **Claude を挟まず Codex へ直接渡した方がよいタスク**
   誤字修正、定型リネーム、明確なテスト追加、機械的な一括置換、依存更新の適用など、**判断が無く証拠も
   機械チェックで足りる**もの。ただし本番/中核に触るなら軽量でも Claude がレビューは通す。

6. **Fable 5 利用中に追加で蓄積しておくべき情報**
   新しい失敗パターン、有効だった指示形、Codex の新しいクセ、環境の地雷、ユーザーの新しい方針を、
   その都度該当ファイルへ追記。`DECISIONS.md` に大きな判断を都度記録。テストフィクスチャ（実挙動の
   スナップショット）を増やす（後継が実データで検証できる資産になる）。

7. **運用が形骸化しているか確認する方法**
   兆候: (a) レビューで実行結果を見ず Codex 報告のまま合格にしている、(b) 変更禁止範囲を書かなくなった、
   (c) 「テスト緑」だけで完了にして実挙動を見ていない、(d) デプロイ後確認を飛ばしている、
   (e) 同じ失敗が `DECISIONS`/例に無いのに繰り返している。
   点検法: 直近の受入 3 件で「どの実行結果を自分の目で見たか」を言えるか自己確認する。言えないなら形骸化。
