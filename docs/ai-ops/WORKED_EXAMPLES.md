# 実例集（このプロジェクトに沿った 8 パターン）

各例は「悪い指示 → 問題点 → 改善後の指示 → 期待する Codex 出力 → Claude の受入判定」で示す。
多くは 2026-07-18〜19 の実運用から。

---

## 例1: 小さな UI 修正（軽量）

- **悪い指示**: 「モバイルで余白がくっつきすぎだから直して」
- **問題点**: 対象要素が不定。どの要素間の余白か分からず、別要素を触って外す（実際に外した）。
  さらに 2px の変更では見た目が変わらず「変わらん」になる。
- **改善後**: 「対象 `app.js`。モバイル（`@media(max-width:780px)`）で、トップバー検索欄 `.top-search` と
  直下の `.overview-hero` の**縦の間隔**を、他セクション間と同じ約 16px に揃える。他要素は触るな。
  完了条件: `node --check app.js`。変更箇所の CSS を報告。」
- **期待する Codex 出力**: 該当メディアクエリ内の margin/gap を 1 箇所変更、before/after を提示。
- **受入判定**: `git diff` で対象要素だけか確認 → 本番 HTML を `curl` して該当セクション間隔を目視。
  **対象要素が合っているか**を最優先で見る（コードが動く≠意図一致）。

## 例2: API 仕様変更（通常〜高リスク）

- **悪い指示**: 「結果 API に infraFailure を足して」
- **問題点**: 受信側（app.js の正規化・保存）と送信側（dispatcher）の両方に整合が要るのに片方だけ変わる。
- **改善後**: テンプレ B で。必須要件「dispatcher が result POST に `infraFailure` を送る / app.js が
  受理して attempts 返金 & infraRetries++」、非要件「done/deferred の既存挙動は変えない」、
  変更禁止「他ステータス分岐」、検証「`tests/infra-failure-requeue.check.js` 追加して緑」。
- **期待する Codex 出力**: 両側の差分 + 新テスト + `npm run check` 緑。
- **受入判定**: 自分で `npm run check`。さらに実ジョブで `infraRetries` が増える経路をログで確認。

## 例3: 既存機能を壊しやすい変更（高リスク）

- **悪い指示**: 「受信を WS デコードに置き換えて」
- **問題点**: 本番デーモンの中核。全面置換は既存正常系を壊す。
- **改善後**: テンプレ C。段階導入必須（`PSEUDO_CODEX_WS_CAPTURE` off/shadow/on、既定 off は不変）、
  フォールバック必須（WS 空なら DOM）、「off の挙動は 1 バイトも変えるな」。
- **期待する Codex 出力**: フラグ分岐 + シャドー比較ログ + DOM フォールバック。
- **受入判定**: off で実ジョブ無回帰 → shadow で WS と DOM のバイト一致をログ確認 → on で実 E2E 完走 →
  デプロイ後も配備物に統合が残ることを確認。**段階ごとに実挙動の証拠**。

## 例4: 調査だけ先にやらせる

- **悪い指示**: 「なんで応答が途中で切れるのか直して」
- **問題点**: 原因未特定のまま実装させると的外れ。
- **改善後**: テンプレ D。「実装するな。フィクスチャ `tests/fixtures/ws-conversation-sample.json` を
  調べ、1 ターンに `finished_successfully` が何回出るか、`message_stream_complete` は何回か、
  end_turn の有無を数えて報告せよ。」
- **期待する Codex 出力**: `finished_successfully: 94` 等の実測値のみ。変更なし。
- **受入判定**: 数字が実データ由来か（Claude 自身も `grep`/`node -e` で裏取り）。この結果から**方針を選ぶ**
  （＝早期 done ポーリングをやめ、DOM 完了後に一括デコードする、を Claude が決定）。

## 例5: 実装後に証拠不足で差し戻す

- **悪い指示なし。Codex の報告**: 「`WS_DECODE_OK` を確認済みです。」
- **問題点**: 自己申告のみ。Claude が信じずに再実行。
- **Claude の行動**: 自分で `node tests/ws-decode.check.js` を回す。さらにテストが**本物か**（domTruth と
  厳密比較しているか、通すために緩めていないか）テスト本文を読む。カバレッジ不足（1 ケースのみ）を発見し、
  コードブロックのフィクスチャを追加採取して再検証。
- **受入判定**: 「証拠不足」→ 追加ケースで再検証して初めて「合格」。

## 例6: Codex がユーザー意図と違う実装をした

- **現象**: 「on モードのバグを直せ（DOM 完了後に全フレームをデコードする形に変えろ）」と指示したら、
  Codex は WS 統合を**丸ごと削除**して DOM のみに revert した。
- **問題点**: 「fix」を「機能を消す」と誤解。変更禁止範囲を書いていなかった。
- **改善後の対応**: 差分を `grep` で確認して「機能が全消し」と即検出 → **らちが明かないと判断し Claude が
  自分で実装**（4 箇所の Edit を正確に適用）。再委任に固執しない。
- **受入判定**: 「仕様違反（削除）」→ Claude 実装 → 自分で構文/テスト/実 E2E で確認。

## 例7: デプロイまで必要なタスク

- **改善後の指示 + フロー**: `agent/<機能名>` ブランチ → 変更 → **`deploy/deploy-request-console` の
  manifest に新規ファイルを追加** → `package.json` にテスト追記 → `npm run check` 緑 → PR → merge →
  `scripts/deploy-request-console-main`。
- **受入判定**: `REQUEST_CONSOLE_RELEASE_OK` + 配備物（`/home/ubuntu/chatgpt-browser-agent/`）に統合が
  入っているか `grep` → デプロイ後の実ジョブ完走。**「デプロイで消えないか」を必ず確認**（ADR-005）。

## 例8: 一度失敗したコマンドを別戦略で復旧

- **現象**: 旧い失敗ジョブを再実行しても毎回 `GIT_REBASE_CONFLICT` で死ぬ（ブランチが古く base と衝突）。
- **悪い対応**: 同じ再実行を繰り返す（同じ結果）。
- **改善後**: 原因（古いジョブブランチが最新 main と衝突）を特定 → dispatcher に「worktree 再利用時に
  最新へ自動 rebase、衝突時は archive ブランチへ退避して base から仕切り直し」を実装（テンプレ B、
  テストで衝突ケースを再現）→ デプロイ → 実際に古い失敗ジョブを再実行して attempt=1 完走を確認。
- **教訓**: 一度失敗したコマンドを同じ形で繰り返すな。**原因を特定してから戦略を変える**。

## 共通する Claude 側の勝ちパターン

- 対象を要素/ファイル/行まで特定してから投げる。
- 「触るな」を必ず書く。
- 段階導入（フラグ/シャドー）で既定を守る。
- Codex の「できた」を自分で再実行して確かめる。テストが本物かも読む。
- らちが明かない中核は自分で実装する。
- デプロイで消えないか、次の同操作で再現するかを最後に確認する。
