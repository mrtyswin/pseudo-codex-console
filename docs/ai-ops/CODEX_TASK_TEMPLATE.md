# Codex タスクテンプレート

Claude が Codex へ実装を委任するときに使う。**空欄を残さない**。空欄は Codex が自分の解釈で埋め、
意図とズレる。判断が要る欄は、Claude が判断した**結論**を書く（Codex に判断させない）。

このプロジェクトでの Codex 起動方法（非対話・実運用実績あり）:

```bash
codex exec --dangerously-bypass-approvals-and-sandbox --skip-git-repo-check \
  -C /home/ubuntu/chatgpt-projects/request-console - < brief.md
```

（`- < brief.md` で brief を stdin から渡す。`-C` は作業ディレクトリ。危険フラグは外部サンドボックス
前提の本ホストでのみ。他環境では `-s workspace-write` 等に落とすこと。）

---

## テンプレA：軽量タスク（1 ファイル・低リスク）

```
タスク: <1 行で>
対象ファイル: <パス>（他は触るな）
やること: <具体的に。before/after が言えるなら書く>
やらないこと: <近接する変えてはいけない箇所>
完了条件: <node --check <file> が通る / 該当テスト XXX が通る>
報告: 変更した箇所と node --check 結果だけ書け。
```

---

## テンプレB：標準タスク（機能追加・複数ファイル）

```
# 背景（事実。信頼してよい前提）
<Claude が調査・固定した事実。ここが薄いと Codex は迷う>

# 目的
<この変更で解決したいこと。1〜2 文>

# ユーザーの意図
<言葉どおりでなく、本当に欲しい結果>

# 現在の状態
<今どうなっているか。関連ファイル・関数・行のアンカー>

# 必須要件
- <満たすべきこと>

# 非要件（やらないこと）
- <過剰実装を止める。ここが効く>

# 変更可能範囲
- <触ってよいファイル/領域>

# 変更禁止範囲
- <絶対に触るな。既存の正常系・無関係ファイル・公開 API など>

# 調査対象（必要なら先に読め）
- <Codex がまず把握すべき箇所>

# 実装方針（Claude が選んだ結論）
<複数案から Claude が選んだ方針。Codex に選ばせない>

# 完了条件
- <機能面での合格条件>

# 検証コマンド（Codex 自身が実行して通す）
- node --check <...>
- node tests/<...>.check.js
- npm run check   # 全体に影響する時

# 必須の証拠（報告に含めよ）
- 追加/変更したファイルと要点
- 検証コマンドの実行結果（出力の該当行）

# 失敗時の行動
- 通らないなら原因を述べ、実データ（node -e 等）で確認し、合うまで直す。
  一度失敗したコマンドを同じ形で繰り返すな。戦略を変えろ。

# 出力形式
- 変更ファイル / 検証結果 / 実装の要点 3〜5 行。
```

## テンプレC：高リスク変更（既存を壊しうる／本番／中核）

テンプレB に加えて：

```
# 段階導入
- 環境変数/フラグでオプトインにせよ。既定（フラグ無し）の挙動は 1 バイトも変えるな。
- 可能ならシャドーモード（新旧を並走させ結果をログ比較。採用は旧のまま）を先に用意せよ。

# 既存正常系の保護
- off/既定の経路が従来と完全一致であることを、コードと挙動の両方で確認して報告せよ。

# フォールバック
- 新経路が失敗/空の時、旧経路へ自動フォールバックせよ（最悪でも従来と同等）。

# ロールバック手段
- 問題時にどう戻すか（フラグ off / revert / 再デプロイ）を報告に明記せよ。
```

## 「調査だけ」を先にやらせるテンプレD

重い/不明なタスクは、実装前に調査だけ分離する（実装を急がせない）。

```
# 調査タスク（実装するな。読んで報告だけ）
目的: <何を知りたいか>
調べる対象: <ファイル/挙動/データ>
報告してほしい事実:
- <質問 1>
- <質問 2>
禁止: この段階でファイルを変更するな。事実だけ返せ。推測は「推測」と明記せよ。
```

Claude はこの調査結果を見て**実装方針を選び**、改めてテンプレ B/C を投げる。

---

## テンプレを使うときの Claude 側の注意

- **アンカーを与える**: 行番号・関数名・既存文字列を書くと Codex が正確に当てられる。
- **「触るな」を必ず書く**: 本プロジェクトで、修正指示を「機能ごと削除」と誤解された実例がある。
  変更禁止範囲を明記すれば防げる。
- **証拠を要求する**: 「通ることを自分で確認してから報告」と必ず入れる。Claude 側でも**再実行して**確かめる。
- **1 タスク 1 目的**: 複数目的を混ぜると受入判定が曖昧になる。
