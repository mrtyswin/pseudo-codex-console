# AI 上流リーダー運用資産（Claude × Codex）

このフォルダは、**Claude を上流リーダー、Codex を下流の実装担当**として開発を進める運用を、
モデルが変わっても（Fable 5 → Opus / Sonnet / 後継）維持するための再利用可能な資産一式です。

個別の開発タスクの記録ではなく、「**どう働くか**」を明文化したものです。会話履歴が消えても、
このフォルダだけで後継モデルが上流リーダーとして機能できることを目的にしています。

## 読む順（後継 Claude が最初にやること）

1. `MODEL_HANDOFF.md` の冒頭「30 秒サマリー」— まず全体像。
2. `AI_LEADER_GUIDE.md` — 役割と標準フロー。**これが行動規範**。
3. `PROJECT_CONTEXT.md` — 触る前にプロジェクトの前提と地雷を把握。
4. タスクが来たら `CODEX_TASK_TEMPLATE.md` を開いて指示を組む。
5. Codex の結果が来たら `CODEX_REVIEW_CHECKLIST.md` で受入判定。
6. 迷ったら `WORKED_EXAMPLES.md` に同型の実例がある。
7. 大きな設計判断をしたら `DECISIONS.md` に 1 件追記。

軽量タスク（誤字・1 行 CSS）はこの全部を読む必要はない。`AI_LEADER_GUIDE.md` の
「運用レベル」表だけ見て、軽量レベルの手順で進めてよい。

## タスク種別ごとの使い方（早見）

- **軽量**（1 ファイル・低リスク）: テンプレの軽量版で Codex へ → `node --check` / 該当テストだけ確認 → 受入。
- **通常**（機能追加・複数ファイル）: 標準テンプレ → `npm run check` 全通過 + 実挙動 1 つ → 受入。
- **高リスク**（既存を壊しうる・本番デプロイ・添付/送信/受信の中核）: 標準テンプレ + 事前調査タスクを分離 → シャドー/フラグで段階導入 → 実 E2E の証拠必須 → 受入。

## 更新方法（形骸化させないため）

- **いつ更新するか**: 新しい失敗パターン、有効だった指示形、Codex のクセ、環境の地雷を見つけたら**その場で**該当ファイルへ 1〜3 行追記する。まとめて後回しにしない。
- **古い情報の扱い**: 事実が変わったら該当行を書き換える。日付を添える（相対日付でなく絶対日付）。判断記録（DECISIONS）は消さず「却下/上書き」を追記する。
- **重複禁止**: 同じ内容を複数ファイルに書かない。事実は 1 か所（多くは PROJECT_CONTEXT か DECISIONS）に置き、他はリンク参照。
- **矛盾チェック**: 大きめの更新後、`AI_LEADER_GUIDE` と `PROJECT_CONTEXT` と `DECISIONS` の間で食い違いがないか一度読み返す。

## この資産の由来

Fable 5 が上流リーダー、Codex 5.6 が実装担当として本プロジェクト（Pseudo Codex Console /
ブラウザ版 ChatGPT を疑似 Codex 化するジョブキュー）を開発した実運用から抽出した。
抽象論でなく、実際に起きた成功・失敗・判断を根拠にしている。推測で埋めた箇所は
各ファイル内で「（未確認）」と明記している。
