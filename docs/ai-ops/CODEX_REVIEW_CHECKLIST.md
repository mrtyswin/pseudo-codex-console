# Codex 成果物レビュー・チェックリスト

Codex が「完了しました」と言った後、Claude が受入可能かを判定する。
**Codex の説明文でなく、実際の証拠で判定する。**

---

## 判定ラベル（どれか 1 つに分類）

| ラベル | 意味 | 次の行動 |
|---|---|---|
| 合格 | 意図を満たし、証拠も揃っている | 受入。必要なら本番へ |
| 修正が必要 | 方向は合っているが不足がある | 不足点を具体的に再指示 |
| 証拠不足 | 主張はあるが実行結果がない | 証拠（diff/テスト/ログ）を要求、または自分で実行 |
| 仕様違反 | 変更禁止範囲を触った/要件を外した | 該当を戻させ、範囲を明示して再指示 |
| 未完了 | 途中で止まっている | 続きを指示、または Claude が引き取る |
| 実装は正しいが意図と不一致 | コードは動くがユーザーが欲しいものと違う | 目的を再固定して方針から出し直す |

---

## 受入前に自分で確認する証拠（自己申告を信じない）

該当するものを**自分で実行**して観測する:

- [ ] `git status` / `git diff` — 触ったファイルは想定どおりか。**変更禁止範囲に手が入っていないか**。
      余計な差分（無関係な整形・削除）はないか。
- [ ] 構文: `node --check <file>`（Python は `python3 -m py_compile`）。
- [ ] テスト: 該当 `node tests/<...>.check.js`、影響が広いなら `npm run check` 全体。**自分で回して緑を見る**。
- [ ] lint / ビルド（あれば）。
- [ ] 実挙動（最重要）: 実際に動かして観測する。本プロジェクトなら
      - 実ジョブを 1 本流して完走を確認（`curl` 投入 → 状態ポーリング → finalAnswer 確認）
      - ログ確認（`grep` で期待の経路・値が出ているか）
      - 本番 HTTP / HTML（`curl http://127.0.0.1:8090/...`）
      - デプロイ結果（`AUTO_DEPLOY_OK` / 本番ファイルの差分ゼロ）
- [ ] 永続性: デプロイ/再起動で消えないか（正典に入っているか、デプロイ manifest に載っているか）。

**「テストだけ緑」で合格にしない。** ユーザーが実際にやる操作で 1 経路は実挙動を見る。

---

## 意図一致の確認（コードの正しさとは別軸）

- [ ] ユーザーの**本当の目的**（Claude が 1 文で固定したもの）を満たすか。
- [ ] 対象要素は合っているか（「余白」なら**その要素**の余白か。別要素を触っていないか）。
- [ ] 非要件を侵していないか（過剰実装していないか）。
- [ ] 既存の正常系挙動が変わっていないか。

コードが動く ≠ 意図を満たす。この 2 つを必ず分けて判定する。

---

## 差し戻し（修正が必要）の書き方

悪い: 「動かないので直して」。
良い:
```
現象: <何がどう不足/違うか>
証拠: <どのコマンド出力/ログでそう判断したか>
原因: <分かっているなら>
触ってよい範囲: <...>
触るな: <...>
完了の証拠: <再度満たすべきもの>
```

---

## 「合格」を出す前の最終自問

1. 実行結果を**自分の目で**見たか？（見ていないなら「証拠不足」）
2. これはデプロイ/再起動/次の同操作で**再現するか**？
3. ユーザーが今から同じことをして、**同じ良い結果**になるか？
4. 「できた」と言って、後で覆らないと**言い切れるか**？

1 つでも No/不明なら、合格にしない。
